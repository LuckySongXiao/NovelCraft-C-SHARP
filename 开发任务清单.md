# 小说管理系统开发任务清单

## 1. 文档说明

本文档基于 `优化实施清单.md` 继续细化，目标是将优化方案进一步拆解成可以直接进入开发的任务包。

本文档面向以下使用场景：

- 作为阶段开发排期清单
- 作为实际编码时的任务切分依据
- 作为多人协作时的分工参考
- 作为验收和阶段复盘的核对清单

## 2. 开发目标

本轮开发的总目标是：

1. 先打通真实业务主链路
2. 再统一导航、上下文、AI 和报表能力
3. 最后再进行结构优化与测试补强

主链路定义如下：

`项目选择 -> 项目概览 -> 卷章/角色/剧情/设定 -> AI协作 -> 导入导出 -> 统计分析`

## 3. 执行策略

### 3.1 总体策略

- 先修复会导致“项目错乱”“功能断链”“数据不一致”的问题
- 再处理重复逻辑、导航机制、上下文拼装等结构问题
- 最后处理页面拆分、测试补强和代码质量提升

### 3.2 任务优先级定义

- `P0`：必须优先处理，否则主流程不稳定
- `P1`：应尽快处理，否则会阻碍后续统一改造
- `P2`：优化质量和扩展性，允许稍后安排
- `P3`：增强项，可放到后续迭代

### 3.3 状态定义

- `未开始`
- `进行中`
- `待联调`
- `待验收`
- `已完成`
- `延期`

## 5. 第一阶段详细任务清单

## T01 清理随机项目 ID 和固定项目 ID

### 目标

消除当前项目上下文不一致的主要来源。

### 问题表现

- 某些页面直接使用固定项目 ID
- 某些功能使用 `Guid.NewGuid()` 作为当前项目 ID
- 这会造成不同功能看到的并非同一个项目

### 重点排查位置

- `src/NovelManagement.WPF/Views/MainWindow.xaml.cs`
- `src/NovelManagement.WPF/Views/ProjectOverviewView.xaml.cs`
- `src/NovelManagement.WPF/Views/ImportExportView.xaml.cs`
- `src/NovelManagement.WPF/Views/AIOutlineGeneratorDialog.xaml.cs`
- `src/NovelManagement.WPF/Views/PrerequisiteGenerationDialog.xaml.cs`

### 实施动作

1. 搜索所有作为项目上下文使用的 `Guid.NewGuid()`
2. 搜索固定项目 ID 常量
3. 将其替换为统一来源：
   - `ProjectContextService.CurrentProjectId`
4. 若当前项目不存在：
   - 不再静默伪造项目
   - 改为提示用户先选择项目

### 验收标准

- 任意页面进入后使用的项目 ID 可追溯到 `ProjectContextService`
- 主流程中不再出现随机项目 ID

### 依赖

- 无

### 风险

- 某些页面依赖旧的演示逻辑，替换后可能出现空数据

### 建议状态

- `立即执行`

## T02 移除默认项目自动创建兜底

### 目标

避免页面在无项目状态下偷偷制造数据或切换项目。

### 问题表现

- 角色页、章节页等在没有当前项目时会尝试自动创建“默认项目”
- 这种做法会污染真实数据，并掩盖主流程设计问题

### 重点排查位置

- `src/NovelManagement.WPF/Views/CharacterManagementView.xaml.cs`
- `src/NovelManagement.WPF/Views/ChapterEditorDialog.xaml.cs`

### 实施动作

1. 删除页面级默认项目创建逻辑
2. 替换为统一的“未选择项目”处理流程
3. 增加空状态提示：
   - 提示选择项目
   - 提供跳转到项目管理页的入口

### 验收标准

- 无项目状态下，页面不再自动创建项目
- 页面显示明确提示，而不是默默进入伪项目

### 依赖

- T03

### 风险

- 页面空状态体验如果没有处理好，用户会误以为功能失效

### 建议状态

- `立即执行`

## T03 建立统一当前项目校验机制

### 目标

让所有需要项目上下文的功能都有统一入口校验。

### 建议方案

新增统一帮助器或服务，例如：

- `CurrentProjectGuard`
- `ProjectAccessService`

### 建议职责

- 判断当前是否已选择项目
- 获取当前项目 ID
- 若未选择项目，统一抛出友好提示或返回失败结果

### 实施动作

1. 定义统一校验方法
2. 替换页面内各自判断逻辑
3. 将以下入口统一纳入校验：
   - 角色管理
   - 关系网络
   - 剧情管理
   - 卷章管理
   - AI 大纲生成
   - 前置条件生成
   - 导入导出
   - 统计分析

### 涉及文件

- `src/NovelManagement.WPF/Services/ProjectContextService.cs`
- `src/NovelManagement.WPF/Views/**/*.xaml.cs`

### 验收标准

- 所有核心业务页都通过统一逻辑获取当前项目
- 页面中不再重复书写大量“当前项目为空怎么办”的逻辑

### 依赖

- 无

### 风险

- 一次改动入口较多，建议按模块分批接入

### 建议状态

- `立即执行`

## T04 概览页接入真实当前项目

### 目标

让概览页从“演示导航面板”变成“当前项目工作台入口”。

### 问题表现

- AI 大纲生成使用随机项目 ID
- 项目设置使用模拟项目对象
- 统计分析展示静态示例数据
- 多数操作只是弹提示或打开独立窗口

### 重点文件

- `src/NovelManagement.WPF/Views/ProjectOverviewView.xaml.cs`

### 实施动作

1. 接入真实项目上下文
2. 将项目设置按钮改为读取当前真实项目数据
3. 将 AI 大纲入口改为使用真实项目 ID
4. 将统计入口改为接入真实统计服务或先接入预留服务
5. 尽量减少独立临时窗口，优先纳入主工作台

### 验收标准

- 概览页所有入口都围绕同一个当前项目工作
- 不再使用随机项目 ID 和模拟项目对象

### 依赖

- T01
- T03

### 风险

- 当前概览页可能承担过多跳转职责，需要适度收敛

### 建议状态

- `立即执行`

## T05 导入导出接入真实项目上下文

### 目标

把导入导出从演示能力转为真实业务功能。

### 问题表现

- 使用模拟项目 ID
- 使用模拟 `UnitOfWork`
- 未通过 DI 获取真实服务

### 重点文件

- `src/NovelManagement.WPF/Views/ImportExportView.xaml.cs`
- `src/NovelManagement.Application/Services/ImportService.cs`
- `src/NovelManagement.Application/Services/ExportService.cs`

### 实施动作

1. 移除页面内模拟服务创建逻辑
2. 改为从 DI 获取真实服务
3. 导入导出统一基于当前项目进行
4. 处理导出历史和导入历史的真实数据来源
5. 完成导入后的联动刷新

### 验收标准

- 当前项目导出可用
- 当前项目导入可用
- 不再使用模拟服务

### 依赖

- T01
- T03

### 风险

- 真实服务接入后会暴露之前被模拟逻辑掩盖的问题

### 建议状态

- `第一批重点任务`

## T06 AI 上下文拼装逻辑合并

### 目标

消除多个 AI 入口重复拼装项目上下文的问题。

### 问题表现

- `AIChapterWriteDialog`
- `AIContinueWriteDialog`

两个入口中存在重复的 `GetProjectContextDataAsync()` 逻辑。

### 建议新增组件

- `src/NovelManagement.WPF/Services/ProjectContextAssembler.cs`

### 实施动作

1. 提取统一 AI 上下文装配服务
2. 统一返回结构：
   - 项目信息
   - 剧情摘要
   - 角色摘要
   - 世界设定摘要
   - 可扩展字段
3. 将两个 AI 对话框改为调用统一服务
4. 为后续扩展时间线、势力、章节摘要预留结构

### 验收标准

- 两个 AI 入口使用同一套上下文生成逻辑
- 后续新增上下文维度时只需改一个地方

### 依赖

- T03
- T09

### 风险

- 如果装配逻辑设计得太死，后续 AI 功能扩展会受限

### 建议状态

- `第一批预研，第二批落地`

## T07 AICollaborationView 使用统一工作流

### 目标

让 AI 协作页面不再维护一套独立于应用容器之外的运行逻辑。

### 问题表现

- 页面内手工创建 `TaskQueue`
- 页面内手工创建 `NovelWorkflowEngine`
- 服务缺失时又创建本地替代对象

### 重点文件

- `src/NovelManagement.WPF/Views/AICollaborationView.xaml.cs`
- `src/NovelManagement.WPF/App.xaml.cs`
- `src/NovelManagement.AI/Extensions/ServiceCollectionExtensions.cs`

### 实施动作

1. 明确 AI 服务和工作流的单一注册入口
2. 页面只获取，不自行创建核心对象
3. 清理重复注册逻辑
4. 补齐缺失的 provider 接入点

### 验收标准

- AI 协作页面不再 `new` 核心工作流对象
- 应用内只保留一套有效工作流实例来源

### 依赖

- T06

### 风险

- 页面目前可能依赖本地对象初始化顺序，需要联调验证

### 建议状态

- `第一批预研，第二批落地`

## 13. 结论

这份开发任务清单的目的，是把“优化方向”进一步变成“可直接开工的任务包”。

当前最值得马上推进的不是全面重构，而是完成以下五项：

- 清理随机和固定项目 ID
- 移除默认项目自动创建
- 建立统一当前项目校验
- 概览页接入真实项目
- 导入导出接入真实项目

只要这五项完成，项目会从“功能很多但联动脆弱”，进入“主链路稳定、后续可持续优化”的状态。
