# 小说管理系统优化实施清单

## 1. 文档目的

本文档用于将现阶段对项目功能联动短板的分析，转化为可执行、可验收、可排期的优化实施方案。

本文档重点解决以下问题：

- 功能很多，但跨模块主流程不稳定
- 项目上下文未成为所有功能的统一入口
- AI、导入导出、统计分析存在演示化实现，尚未形成生产级闭环
- WPF 页面之间联动依赖手工代码，缺乏统一导航、刷新和事件机制
- 代码分层设计与实际落地存在偏差，维护成本持续增大

## 2. 当前阶段核心判断

### 2.1 当前系统的主要矛盾

当前项目的核心问题不是单个功能缺失，而是以下三个方面没有形成统一主链路：

1. 当前项目上下文没有被强制纳入所有功能入口
2. 页面导航与模块刷新没有统一标准
3. AI、导入导出、统计等能力各自成岛，不能围绕同一项目数据闭环

### 2.2 优化目标

本轮优化的目标不是大范围重写，而是先建立一个稳定的、可扩展的业务主链路：

`项目选择 -> 项目概览 -> 卷章/角色/剧情/设定 -> AI协作 -> 导入导出 -> 统计分析`

达到以下效果：

- 用户进入任意模块时，都明确隶属于当前项目
- 任意模块修改数据后，相关模块能够刷新或收到通知
- AI 获取到的是统一、完整、可复用的项目上下文
- 导入导出和统计分析使用真实项目数据，而非模拟数据
- 后续再做 MVVM 拆分时，有稳定基础，不会边改边塌

## 3. 现状问题清单

## 3.1 项目上下文问题

- `ProjectContextService` 已存在，但没有成为全局强制依赖
- 多个页面在没有当前项目时自行创建“默认项目”
- 部分入口直接使用固定 `Guid` 或 `Guid.NewGuid()`
- 不同功能打开时，项目身份可能不一致

### 典型位置

- `src/NovelManagement.WPF/Services/ProjectContextService.cs`
- `src/NovelManagement.WPF/Views/CharacterManagementView.xaml.cs`
- `src/NovelManagement.WPF/Views/ChapterEditorDialog.xaml.cs`
- `src/NovelManagement.WPF/Views/ProjectOverviewView.xaml.cs`
- `src/NovelManagement.WPF/Views/MainWindow.xaml.cs`

## 3.2 页面导航与生命周期问题

- `MainWindow` 通过 `new XxxView()` 直接切换页面
- 页面依赖 `App.ServiceProvider` 自行取服务，依赖关系隐藏
- 大多数页面没有统一的进入、退出、刷新接口
- 页面之间联动依赖特判代码，而不是统一机制

### 典型位置

- `src/NovelManagement.WPF/Views/ProjectOverviewView.xaml.cs`

- 角色管理与关系网络之间只有局部手工联动
- 关系网络未扩展为统一知识网络
- 剧情、势力、时间线、世界设定之间缺少统一读模型
- 数据引用关系没有统一抽象，后续联动成本高

### 典型位置

## 4. 优化实施原则

为避免重构范围失控，本次实施遵循以下原则：

### 4.1 先打通主链路，再做大重构

优先保证真实业务流程可用，再逐步做 MVVM、导航框架、事件总线等结构升级。

### 4.2 优先消除伪联动

凡是以下场景，优先整改：

- 随机项目 ID
- 固定默认项目 ID
- 模拟数据自动回退
- 模拟服务替代真实服务
- 页面手工同步其他页面数据

### 4.3 统一单一事实来源

以下信息必须只有一个主来源：

- 当前项目
- 项目聚合上下文
- AI 工作流实例
- 导入导出服务实例
- 统计数据聚合入口

### 4.4 改造时保留可运行状态

每个阶段结束后，系统都必须保持：

- 可编译
- 主流程可运行
- 至少有一条跨模块闭环可验证

## 5. 优化目标架构

建议逐步形成以下结构：

### 5.1 上下文层

- `ProjectContextService`
  - 负责当前项目身份
  - 提供当前项目变更事件
  - 不再负责兜底创建默认项目

- `ProjectReadModelService`
  - 聚合项目概览、角色、剧情、卷章、设定、统计的只读数据
  - 为概览页、AI、导出、统计提供统一读模型

- `ProjectContextAssembler`
  - 专门为 AI 功能装配上下文
  - 统一项目、卷、章节、剧情、角色、设定、时间线等信息

### 5.2 导航层

- `INavigationService`
  - 负责页面切换
  - 负责导航参数
  - 负责页面生命周期回调

- `IProjectAwareView`
  - 定义项目切换后的刷新行为

- `IRefreshableView`
  - 定义页面进入和数据更新时的刷新行为

### 5.3 事件联动层

- `DomainEvent` 或轻量级 `AppEvent`
  - 用于角色更新、剧情更新、章节保存、设定更新、AI结果生成完成

- `EventDispatcher`
  - 统一页面内和服务层的事件派发

### 5.4 AI 能力层

- `AIAssistantService`
  - 保留为 UI 侧总门面
  - 不再在各页面复制拼上下文逻辑

- `ProjectContextAssembler`
  - 统一 AI 输入上下文

- `AIResultApplyService`
  - 负责把 AI 生成结果回写到章节、剧情、角色、设定

### 5.5 报表与导出层

- `ProjectStatisticsService`
  - 提供项目统计、创作统计、结构统计、AI 使用统计聚合

- `ProjectExportFacade`
  - 基于当前项目驱动导出
  - 不允许页面自行拼装模拟请求

## 6. 分阶段实施计划

## 阶段 0：基线整理与风险隔离

### 目标

建立改造基线，先找出所有伪联动点，避免后续一边改一边引入新的随机逻辑。

### 主要任务

1. 搜索并整理所有以下代码模式：
   - `Guid.NewGuid()` 作为当前项目 ID 使用
   - 固定项目 ID
   - “默认项目”自动创建逻辑
   - `Mock` 服务或模拟数据自动回退
   - 页面内 `App.ServiceProvider` 直接拉取服务
2. 建立整改台账，分为：
   - 必改
   - 可延期
   - 暂保留但需加注释
3. 将演示模式逻辑标记为显式模式，不再静默触发

### 涉及文件

- `src/NovelManagement.WPF/**/*.cs`
- `src/NovelManagement.Application/**/*.cs`
- `src/NovelManagement.AI/**/*.cs`

### 交付物

- 伪联动清单
- 改造优先级清单
- 统一命名规范说明

### 验收标准

- 已标记所有使用随机或固定项目 ID 的功能入口
- 已确认哪些模块仍在使用模拟服务

### 预估工期

2 至 3 天

## 阶段 1：统一项目上下文主链路

### 目标

确保所有核心功能都围绕真实当前项目工作。

### 主要任务

1. 规范 `ProjectContextService` 的职责
   - 仅负责保存、切换、清除当前项目
   - 不负责兜底造项目
2. 在项目选择或项目管理入口处，统一设置当前项目
3. 页面进入前统一校验 `CurrentProjectId`
4. 删除或替换以下行为：
   - 页面内自动创建默认项目
   - 固定项目 ID
   - `Guid.NewGuid()` 充当当前项目 ID
5. 将以下模块统一接入当前项目：
   - 项目概览
   - 卷章管理
   - 角色管理
   - 关系网络
   - 势力管理
   - 剧情管理
   - AI 大纲生成
   - 前置条件生成
   - 导入导出

### 涉及文件

- `src/NovelManagement.WPF/Services/ProjectContextService.cs`
- `src/NovelManagement.WPF/Views/MainWindow.xaml.cs`
- `src/NovelManagement.WPF/Views/ProjectManagementView.xaml.cs`
- `src/NovelManagement.WPF/Views/ProjectOverviewView.xaml.cs`
- `src/NovelManagement.WPF/Views/CharacterManagementView.xaml.cs`
- `src/NovelManagement.WPF/Views/RelationshipNetworkView.xaml.cs`
- `src/NovelManagement.WPF/Views/ChapterEditorDialog.xaml.cs`
- `src/NovelManagement.WPF/Views/AIOutlineGeneratorDialog.xaml.cs`
- `src/NovelManagement.WPF/Views/PrerequisiteGenerationDialog.xaml.cs`
- `src/NovelManagement.WPF/Views/ImportExportView.xaml.cs`

### 交付物

- 当前项目强制校验机制
- 页面统一读取真实项目上下文
- 移除高风险默认项目兜底逻辑

### 验收标准

- 用户选择一个项目后，进入任意核心模块都能读取同一项目
- 核心模块中不再出现随机项目 ID
- 概览页、AI入口、导入导出入口均使用真实项目 ID

### 风险

- 部分页面当前依赖模拟数据，切换到真实上下文后可能暴露旧 bug
- 数据为空时，页面可能需要单独处理空状态 UI

5 至 7 天

## 10. 风险清单与应对策略

| 风险 | 说明 | 应对策略 |
|---|---|---|
| 历史代码耦合过深 | 页面中已有大量 UI 与业务逻辑混写 | 先抽主链路服务，不强求一次拆净 |
| 模拟数据移除后暴露空数据场景 | 真实项目初始状态可能没有任何内容 | 设计空状态页与引导页 |
| 项目切换带来页面状态混乱 | 页面缓存后容易显示旧数据 | 建立页面刷新接口和导航参数清理 |
| AI 上下文过大 | 影响性能与调用成本 | 设计摘要层与优先级筛选 |
| 统计口径不统一 | 各页面各算各的 | 统一由统计服务定义口径 |
| 导入导出链路不完整 | 真实数据结构复杂 | 先做 TXT/Markdown/JSON 的稳定闭环，再扩展格式 |

## 11. 验收方法

每完成一个阶段，至少执行以下验证：

### 11.1 功能验证

- 选择项目后进入概览、角色、剧情、AI、导出、统计，确认项目一致
- 修改角色后，关系网络和相关模块能看到更新
- AI 写作和 AI 续写获取到同一份项目上下文
- 导出当前项目成功，导出结果与实际数据一致
- 统计页数据与项目真实数据基本一致

### 11.2 代码验证

- 新增核心服务后，避免继续在页面中复制相同逻辑
- 避免新增随机项目 ID
- 避免新增模拟服务兜底
- 避免继续扩大 `MainWindow` 的职责

### 11.3 回归验证

- 启动应用
- 打开一个项目
- 进入角色管理
- 打开关系网络
- 执行 AI 功能
- 导出项目
- 打开统计报表

以上流程至少应在每个中阶段结束时手工跑通一次。

## 13. 结论

本项目已经具备较完整的功能面，但当前最需要的不是继续扩功能，而是完成一次“围绕当前项目的能力收口”。

只要优先完成以下四件事，系统整体质量会明显提升：

- 统一项目上下文
- 统一导航与刷新
- 统一 AI 上下文
- 真实化导入导出和统计

完成这四项后，再进行 MVVM 化、事件总线化、页面拆分，投入产出会明显更高。
